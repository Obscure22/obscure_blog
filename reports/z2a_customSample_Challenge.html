Introduction

In this post, I will examine the sample provided in the Zero2Automated: Advanced Malware Analysis course, which is introduced when we reach the third chapter of the course. That said, let's get started.
The scenario is the following:

Hi there,

During an ongoing investigation, one of our IR team members managed to locate an unknown sample on an infected machine belonging to one of our clients. We cannot pass that sample onto you currently as we are still analyzing it to determine what data was exfiltrated. However, one of our backend analysts developed a YARA rule based on the malware packer, and we were able to locate a similar binary that seemed to be an earlier version of the sample we're dealing with. Would you be able to take a look at it? We're all hands on deck here, dealing with this situation, and so we are unable to take a look at it ourselves. We're not too sure how much the binary has changed, though developing some automation tools might be a good idea, in case the threat actors behind it start utilizing something like Cutwail to push their samples. I have uploaded the sample alongside this email.

Thanks, and Good Luck!

Static Analysis

Opening the sample with PE Studio allows us to obtain several useful pieces of information about the binary. The first things that can be noticed are that the file is compiled as a 32-bit executable and that it has a high entropy value (7.434). This high entropy may indicate that parts of the file are packed or encrypted. By looking at the Optional Header, it is possible to see that the Image Base is located at 0x400000 and the Entry Point is at the offset 0x1893. The entry point is located inside the .text section. By analyzing the sections, it is possible to notice that the .text section has an entropy of 6.610, while the .rsrc section has an entropy of 7.994. This suggests that the .rsrc section may contain obfuscated or packed data, while the .text section could include cryptographic operations. In fact, in the Resources section we can see a resource called rcdata. Looking at the first 32 bytes, we can notice that the data appears to be obfuscated. In the Libraries section, there is only one library imported: kernel32.dll. In the Imports section, many APIs from kernel32.dll are imported. In particular, LoadLibraryA and GetProcAddress are used to load other libraries and resolve functions at runtime. Also in the Strings section there are many unusual strings, which suggests that some form of encryption or obfuscation is being used.

Static Analysis with IDA

Opening the sample in IDA redirects us to the main function, which appears as shown in the figure below. It can be observed that there are multiple calls to sub_D41300, each receiving obfuscated strings as input. These strings are then passed as arguments to LoadLibraryA and GetProcAddress.

This indicates that the obfuscated strings correspond to the library name and the function name associated with that library, respectively, which are resolved at runtime. By checking the Xrefs, it is possible to see how many times this function is called throughout the code, along with all the corresponding offsets. The figure below shows the function sub_D41300. Let's try to understand how this function works to decrypt the strings. It can be observed that the code first checks whether the input string length is greater than 0. If so, an alphabet is loaded into v12, and its length is determined. Then, an if-else statement is used to select the correct character from the alphabet, which is subsequently used to replace the corresponding character in the string being decrypted. It can be observed that the sample dynamically loads kernel32.dll at runtime, resolving functions such as FindResourceA, LoadResource, SizeofResource, and LockResource. This behavior could have been anticipated, as we previously noticed in PE Studio that the Resources section contains unusual data. These functions are specifically used to extract and access such resource data, which may indicate the presence of a second-stage payload embedded within the resources.

Now we can rename all the functions we have decrypted in order to make the code clearer and easier to understand. Going further down in the code, the following code snippet can be observed, which shows a familiar pattern. It is possible to observe a first loop that iterates 0x100 (256) times, followed by a second loop that also iterates 0x100 times. From this, it can be inferred that the following code snippet corresponds to the initialization stage of the RC4 algorithm. The first loop initializes the substitution box (S-box) with values ranging from 0x00 to 0xFF. The second loop is then used for the key scheduling, swapping values inside the S-box based on the key in order to prepare the keystream used during the encryption or decryption process. After this part of the code, the final stage of the RC4 algorithm can be observed, which is the XOR stage. A loop performs the XOR operation between the generated keystream and the input data. This loop is iterated as many times as the length of the payload that needs to be decrypted, this information can be found in esi register beacause this register is incremented by one at each iteration. As can be seen from the code, several operations are performed on var_108, after which the byte pointed to in the current iteration is moved into eax. This value is then used to perform the decryption through the XOR operation. Therefore, var_108 represents the key. After this decryption, the malware calls the function sub_CB1000, passing var_110 as an argument. It could contain the decrypted payload. Inside this function, several Windows APIs are deobfuscated, including CreateProcessA. After that, the call to CreateProcessA (via call eax) can be observed. It is also possible to notice that the argument corresponding to dwCreationFlags is set to 0x4. This means that the process (whose handle is stored in ECX) will be created in a suspended state. The sample then call VirtualAlloc to allocate a region of memory to itself with read and write permissions. The result of the VirtualAlloc is the base address of the memory allocated. This base address is passed as an argument to the GetThreadContext function. Then the malware calls WriteProcessMemory to write something in this region of memory. After executing this activity, the sample calls VirtualAllocEx, likely in order to execute the second stage through a remote process injection technique. Finally, the sample calls SetThreadContext to modify the context of the thread, and then ResumeThread to resume and execute the thread.

Dynamic Analysis

By moving to xdbg, it is possible to set several breakpoints on the Windows APIs that were previously identified during the static analysis with IDA, such as VirtualAlloc, VirtualAllocEx, WriteProcessMemory, and CreateProcessA. The first breakpoint is reached at VirtualAlloc, where it is possible to observe that the sample allocates 0x15400 bytes of memory. This memory region is allocated starting at the address 0xB50000. After this, the sample performs the decryption of the payload, as previously observed in IDA, using the RC4 algorithm. The decrypted data is written into the memory region that was previously allocated with VirtualAlloc. After that, in the memory dump we can see that a payload has been decrypted, which essentially corresponds to a PE executable. At the next breakpoint, the execution stops at CreateProcessA. Here it is possible to observe two important parameters: lpApplicationName, which contains the path of the sample itself, and dwCreationFlags, which is set to 0x4 (CREATE_SUSPENDED). When the CreateProcessA has been executed, we can see this new process spawned. The next breakpoint is reached at VirtualAllocEx. Here, the malware allocates a region of memory inside the remote process. It is very interesting to observe the parameters in the stack before the execution of the function. The picture below shows the parameters passed to the function. the first is 0x110 which is the process handle that received this allocation. Then the second is the desired starting address for the region of pages that the process allocate which is 0x400000. Then the third parameter is the size of the region of memory to allocate, in this case the sample allocates 0x18000 bytes. By looking at the handles of the current process, it is possible to observe that the value 0x110 corresponds to the child process that was previously created in a suspended state. This confirms that the malware is now interacting with that process in order to allocate memory and inject the payload into it. The next breakpoint is reached at WriteProcessMemory. At this stage, the malware writes the decrypted payload into the memory region that was previously allocated in the remote process using VirtualAllocEx. As done previously, it is possible to observe some important parameters on the stack that are passed as input to WriteProcessMemory. The first parameter is the process handle, identified by the value 0x110, which corresponds to the child process previously created in a suspended state. The third parameter, 0xB50000, represents a pointer to the buffer that contains the malicious payload to be written into the address space of the specified process. Now that we have found the location where the second stage resides, we can simply right-click on the base address in the memory map (0xB50000), select "Follow in Memory Map", and then extract the memory dump as a file.

Static Analysis - Second Stage

Opening the dumped file in IDA, this is the graph view that we obtain. One interesting function that can be identified almost immediately is sub_401210, which in this case takes the value 0x8436F795 as input. Inside this function, there is first a call to LoadLibrary, followed by a call to GetModuleHandle. This indicates that the sample is trying to load a function, most likely corresponding to the encrypted string passed as input. Between these two calls, there is also an interesting call to sub_401660.
